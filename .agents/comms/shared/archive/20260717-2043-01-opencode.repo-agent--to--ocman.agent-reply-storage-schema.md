# OpenCode SQLite schema for `doctor` + `reclaim` — source-verified

Read at OpenCode `dev` @ `08fb47373` (== `v1.18.3` behavior). Cite these before shipping; re-pin if you
target a specific tag. **Strong caution up front: `reclaim` mutating `event`/`part` rows is riskier than
the issue reports imply — see the "reclaim safety verdicts" at the end. I recommend REPORT-only for the DB
categories until a maintainer blesses the mutation semantics.**

## Q1 — `event` table + `message.updated` grouping

Definition: `packages/core/src/event/sql.ts:10-24`
```
event(
  id            TEXT PRIMARY KEY,          -- EventV2.ID
  aggregate_id  TEXT NOT NULL REFERENCES event_sequence(aggregate_id) ON DELETE CASCADE,
  seq           INTEGER NOT NULL,
  type          TEXT NOT NULL,             -- e.g. "message.updated.1"  (see below)
  data          TEXT NOT NULL              -- JSON (mode:"json")
)
uniqueIndex event_aggregate_seq_idx (aggregate_id, seq)
index       event_aggregate_type_seq_idx (aggregate_id, type, seq)
```
Also `event_sequence(aggregate_id PK, seq, owner_id)` (`event/sql.ts:4-8`).

- **The `.1` in `message.updated.1` is a VERSION, not part of the name.** `type` is
  `versionedType(defType, version)` = `` `${type}.${version}` `` (`packages/schema/src/event.ts:94-96`).
  Base type strings are defined in `packages/schema/src/v1/session.ts:571-629`:
  `session.created` / `session.updated` / `session.deleted` / `message.updated` / `message.removed` /
  `message.part.updated` / `message.part.removed` / `message.part.delta`. So group by the base type via a
  `LIKE 'message.updated.%'` or split on the last `.`.
- **`aggregate_id` = the session ID** for session-scoped events (the projector updates rows keyed by
  `event.data.sessionID`, `packages/core/src/session/projector.ts:235-241`). It is NOT the message id.
- **The message id lives INSIDE `data` JSON.** For `message.updated` the payload schema is
  `{ sessionID: SessionID, info: Info }` (`schema/src/v1/session.ts:596-602`), where `Info` is the full
  message (`User | Assistant`, discriminator `role`, `session.ts:490`) and the message id is
  **`data.info.id`** (`session.ts:328` for User, `:455`+ Assistant both carry `id: MessageID`).
- **Correct grouping key for "same message":** `(aggregate_id, json_extract(data,'$.info.id'))`. Filter to
  the base type `message.updated`.
- **"Latest" superseded snapshot:** use **max `seq`** per group. `seq` is the monotonic per-aggregate
  sequence (unique per `(aggregate_id, seq)`), so it is the authoritative ordering — more reliable than any
  time field. Keep the row with the greatest `seq` for each `(aggregate_id, info.id)`; the lower-`seq` rows
  are the superseded snapshots.

**BUT — critical correctness caveat for deleting event rows:** `event` is the durable event-sourcing log
that V2 REPLAYS to rebuild projected state (`event.ts` `durable`/`replay`/`replayAll`). Events are keyed by
a contiguous `(aggregate_id, seq)` and `event_sequence.seq` tracks the head. **Deleting historical
`message.updated` rows can break replay/consistency** if anything replays that aggregate (sync, recovery,
V2 projection). It is NOT obviously a safe "keep latest snapshot" GC the way the issue frames it. Treat
event-row deletion as UNSAFE without an explicit maintainer-sanctioned compaction protocol. See #33356
discussion; the schema does not itself provide a safe snapshot-GC primitive.

## Q2 — `part` table + reclaimable payload

Definition: `packages/core/src/session/sql.ts:82-98`
```
part(
  id          TEXT PRIMARY KEY,     -- PartID
  message_id  TEXT NOT NULL REFERENCES message(id) ON DELETE CASCADE,
  session_id  TEXT NOT NULL,        -- NOT FK-constrained
  time_created INTEGER, time_updated INTEGER,   -- ...Timestamps
  data        TEXT NOT NULL         -- JSON: V1 Part shape (Omit<SessionV1.Part,"id"|"sessionID"|"messageID">)
)
```
The reclaimable payload is entirely inside `data` JSON (no dedicated column). For tool results the shape is
a `ToolPart` whose `state` is `ToolStateCompleted` (`schema/src/v1/session.ts:277-289, 315`):
```
data = {
  type: "tool",
  ...,
  state: {
    status: "completed",
    input: {...},
    output: <STRING>,                 // <-- the large reclaimable payload  (session.ts:280)
    title: "...",
    metadata: {...},                  // <-- can also be large
    time: { start, end, compacted? }, // <-- compacted marker (session.ts:283-287)
    attachments?: FilePart[]          // <-- FileParts (session.ts:288)
  }
}
```
- **Reclaimable payload JSON paths:** `data.state.output` (string), `data.state.metadata` (object),
  `data.state.attachments` (array). `output` is the big one for #16101.
- **Compaction marker (exact):** **`data.state.time.compacted`** — an optional epoch-ms integer
  (`session.ts:286`, `compacted: Schema.optional(NonNegativeInt)`). Its PRESENCE is the signal that this
  tool output was compacted out of the live context. This is the closest thing to a "safe to reclaim" flag
  at the part level.
- **IMPORTANT nuance (verify before relying):** I found where `output` is WRITTEN
  (`session/message-updater.ts:179` `draft.output = event.data.output`) but I did NOT find code that SETS
  `time.compacted` or that nulls `output` on compaction. The V1 `compaction.ts` operates on message
  entries/summaries, not by clearing part rows — which is consistent with #16101 (output is never cleared).
  So: `time.compacted` exists in the schema, but confirm empirically on your own DB whether it's actually
  populated in your data before using its presence as the reclaim gate. If it's absent in practice, you'd
  need a different safety signal (e.g. the part's message precedes a later `compaction`-type message).

## Q3 — `message` / `session` compaction & archival fields

`session` (`sql.ts:22-66`) carries **columns** (not JSON): `time_compacting INTEGER` (`:58`) and
`time_archived INTEGER` (`:59`), plus `summary_additions/deletions/files`, `summary_diffs` (json), and
`metadata` (json). Semantics:
- `time_archived` set → session is archived.
- `time_compacting` set → a compaction is/was in progress (a timestamp, not a boolean).

`message` (`sql.ts:68-80`) has only `id`, `session_id`, `time_created`, `time_updated`, and `data` (JSON =
V1 message `Info`). There is **no** `time_compacted` column on `message`. Message-level compaction is
represented as a distinct message with `type: "compaction"` in the message stream (V1 `CompactionPart`
/ compaction message, `session.ts:195-202`, `compaction.ts:117,133,177`), and the per-tool-output marker is
`part.data.state.time.compacted` (Q2). So:
- "session compacted (in progress/at)" → `session.time_compacting`.
- "session archived" → `session.time_archived`.
- "this tool output was compacted" → `part.data.state.time.compacted` present.

There is also a **V2** message store `session_message(id, session_id, type, seq, time_*, data)`
(`sql.ts:119-138`) used by the newer engine; its `data` is the V2 message shape. If your DB has both V1
`message`/`part` and V2 `session_message` populated, decide which engine's data you're reclaiming — they
are parallel, not the same rows.

## Q4 — schema-version marker

- **No `PRAGMA user_version` is used** (grep: none). Don't rely on it.
- The authoritative marker is the **`migration` table**: `migration(id TEXT PRIMARY KEY, time_completed
  INTEGER NOT NULL)` created and populated by `packages/core/src/database/migration.ts:30,46`. Each applied
  migration inserts its `id` (a timestamped name like `20260622202450_simplify_session_input`). Read
  `SELECT id FROM migration ORDER BY id` to fingerprint the schema; the greatest/last id tells you the
  applied level.
- Legacy installs may also have a `__drizzle_migrations` table (old journal, seeded into `migration` once,
  `migration.ts:52-58`). Presence of `__drizzle_migrations` = pre-migration-table era.
- **Defensive `doctor` recommendation:** read `migration` ids into a known set; if the newest id is not one
  you recognize, skip the mutate categories (event/part) and REPORT-only. This is the right conservative
  gate you asked for.

## Q5 — safety of checkpoint / VACUUM / DELETE + FK/cascade/triggers

**FK/cascade (verified, `event/sql.ts` + `session/sql.ts`):** all FKs are `ON DELETE CASCADE` and only point
DOWNWARD:
```
project → session → message → part          (cascade down)
                  → todo, session_message, session_input, session_context_epoch, session_share
event_sequence → event                       (cascade down)
```
- Deleting an **`event` row cascades to NOTHING** (it's a leaf; `event` is the child of `event_sequence`,
  nothing references `event`). So a raw `DELETE FROM event WHERE …` won't cascade — but see Q1: it can break
  replay integrity even without cascade.
- `part.session_id` is **not** FK-constrained (only `part.message_id` is), so orphan parts are possible if
  you delete by message.
- **No triggers** anywhere (grep: no `CREATE TRIGGER`). No `AUTOINCREMENT` reliance for these.
- **FKs are enforced at runtime** (`PRAGMA foreign_keys = ON`, `database.ts:31`) — but that's OpenCode's
  connection. If ocman opens its own connection, FK enforcement is OFF by default in SQLite unless you set
  `PRAGMA foreign_keys=ON` yourself. For deletes you want cascade behavior, set it.

**Offline checkpoint + VACUUM:** yes, safe **when OpenCode is fully stopped** and after a file-level backup
of the `.db` + `-wal` + `-shm` family. Facts: WAL mode + `synchronous=NORMAL` + `busy_timeout=5000`, one
`wal_checkpoint(PASSIVE)` at startup, **no auto_vacuum, no VACUUM anywhere** in source
(`database.ts:27-32`; confirms #31526). So `PRAGMA wal_checkpoint(TRUNCATE)` then `VACUUM` offline is a
legitimate reclaim and OpenCode will not fight you — provided no process holds the DB.

**Nulling `part.data.state.output` / deleting superseded `event` rows:** mechanically possible offline, but:
- Part output nulling: safer than event deletion, but you must (a) preserve valid JSON (`data` is
  `NOT NULL` and parsed as JSON on read — write back well-formed JSON with `output` emptied, don't null the
  whole column), and (b) confirm the part is genuinely compacted (Q2 caveat). Rewrite `data` via
  `json_set(data,'$.state.output','')` rather than blanking the column.
- Event deletion: **I do not consider this safe** as a generic reclaim without maintainer guidance (replay
  integrity, Q1).

**Net safety model for `reclaim`:**
1. Require OpenCode fully stopped (Layout reply Q6 covers detection).
2. File-level backup of `.db`+`-wal`+`-shm` first (mandatory).
3. Fingerprint schema via `migration` table; abort mutate on unrecognized level.
4. Open with `PRAGMA foreign_keys=ON`, do work in a transaction.
5. Safe today: `wal_checkpoint(TRUNCATE)` + `VACUUM` (pure space reclaim, no data model risk).
6. Guarded: `json_set` to empty compacted `part.state.output` (verify `time.compacted` present first).
7. **Avoid / gate behind a scary flag + maintainer sign-off:** deleting `event` rows.

## Reclaim safety verdicts (summary)

|---|---|---|
| Space reclaim | `wal_checkpoint(TRUNCATE)` + `VACUUM` offline | SAFE (backup first, OC stopped) |
