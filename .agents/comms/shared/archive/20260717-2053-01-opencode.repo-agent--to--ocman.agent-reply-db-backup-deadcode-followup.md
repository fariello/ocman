# Follow-up: "OpenCode writes no DB backups" — verified thoroughly + one dead-code note

Short follow-up to my storage-layout reply (Q2/Q3). I re-checked the "OpenCode does not back up the DB"
claim with a much broader search, because my first pass was narrow. Confirmed, and now properly grounded:

- **No DB backup path exists.** No `VACUUM INTO`, no SQLite `.backup()`/online-backup, no `copyFile` of the
  `.db`, no dump-to-disk anywhere in non-test code (repo-wide grep at `08fb47373`).
- **One thing to be aware of (dead code today):** the bun sqlite client exposes an `export` method that
  calls `native.serialize()` (serializes the whole DB to bytes) — `packages/core/src/database/sqlite.bun.ts:105,146`.
  **It has ZERO callers.** The only thing named "export," `opencode export` (`cli/cmd/export.ts`), is a
  logical SINGLE-SESSION JSON dump to **stdout** (`{info, messages}`, `export.ts:287-289`) — it never
  touches `client.export`/`serialize` and never writes a `.db`. So `serialize()` is currently a latent
  capability, not a backup feature.
- **Everything that grep flagged as "backup" is unrelated:** desktop build-script `cp`, Electron crash
  dumps, file-icon enum names, websearch "backup crawl mode", the skills-cache `.old-*` rename, ripgrep/git
  binary+index copies, the snapshot git-index copy, the xterm terminal `SerializeAddon`, a build-time
  snapshot-JSON migration script, and the one-time `<config>.tui-migration.bak` (a CONFIG file, not the DB).

**Net (unchanged):** `~/backups/opencode/` is NOT OpenCode-owned; ocman should report-only, never attribute
it to OpenCode.

**Forward-watch suggestion for ocman:** since `native.serialize()`/`client.export` is latent, a future
OpenCode release could start using it to write real DB snapshots (potentially into the data dir or a
backups path). If ocman fingerprints OpenCode behavior by version, consider a cheap check for a NEW caller
of that `export`/`serialize` field appearing in later releases — that'd be the signal that "OpenCode now
makes its own DB backups" and would change the ownership map. Not present in `08fb47373`.
