## The big update: `time.compacted` IS populated, and `--reclaim-parts` is provably safe

In my schema reply I said "found no code that sets `time.compacted` or clears output." That was
incomplete — I hadn't found the V1 pruner. Correcting it now, with the exact source:

- **OpenCode DOES set `time.compacted`.** `packages/opencode/src/session/compaction.ts:243-287` (`prune`)
  walks tool parts oldest-first and, for completed non-protected parts beyond a token budget, sets
  `part.state.time.compacted = Date.now()` and `updatePart` (`:281-282`). So the marker is real and
  populated **when the user has `compaction.prune` enabled** — which is an OPTIONAL, DEFAULT-OFF config key
  (`compaction.ts:245` `if (!cfg.compaction?.prune) return`; schema `core/src/v1/config/config.ts:154`).
  → Your VERIFY-OR-SKIP design is exactly correct: some DBs will have populated markers, many won't. Keep
  the empirical probe + fail-closed.
- **Crucially, `prune` sets the marker but does NOT empty `output`** (`:281-282` only stamps the timestamp;
  no `output = ""`). That is precisely issue #16101 — so there genuinely IS reclaimable payload behind a
  populated marker. Good: your reclaim has real work to do and isn't redundant.
- **Emptying `data.state.output` for a compacted part is SAFE — proven by how the request is rebuilt.**
  When OpenCode reconstructs the model request, a compacted completed tool part's output is REPLACED with a
  literal placeholder and its attachments are dropped, WITHOUT reading `state.output`:
  ```
  packages/opencode/src/session/message-v2.ts:293-296
  const outputText = part.state.time.compacted
    ? "[Old tool result content cleared]"
    : truncateToolOutput(part.state.output, ...)
  const attachments = part.state.time.compacted || options?.stripMedia ? [] : (part.state.attachments ?? [])
  ```
  So once `time.compacted` is set, OpenCode itself never re-reads `output` (or `attachments`) for that part
  when talking to the model — it substitutes the placeholder. **Therefore ocman emptying `output` on disk
  changes nothing OpenCode would have sent.** This is the strongest possible confirmation for Q2.

## Answers to your 5 questions

**Q1 — `message.updated` grouping `(aggregate_id, $.info.id)`, latest = max seq, REPORT-ONLY: CORRECT.**
- Base type is `message.updated`; stored `type` is version-suffixed (`message.updated.1` via `versionedType`,
  `packages/schema/src/event.ts:94`). Split on the last `.` or `LIKE 'message.updated.%'` — both fine.
- `aggregate_id` = session id; the message id is at `$.info.id` (payload `{sessionID, info}`,
  `schema/src/v1/session.ts:596-602`; `info` is `User|Assistant`, both carry `id`, `:328/:455`).
- `seq` is the authoritative per-aggregate ordering (unique `(aggregate_id, seq)`), so max `seq` = newest. Good.
- **`$.info.id` absent/renamed?** For `message.updated` events, `info.id` is always present (schema-required).
  Two caveats for a defensive REPORT query: (i) OTHER event types under the same `aggregate_id` won't have
  `$.info.id` — your `LIKE 'message.updated.%'` filter handles that; (ii) `json_extract` returns NULL for
  any row lacking the path, so rows with NULL `info.id` should be excluded from the estimate, not grouped
  together. Since it's report-only you're safe regardless. No rename across the versions I can see.

**Q2 — empty ONLY `data.state.output`, leave metadata/attachments/input/title/time: SAFE (see above).**
- Confirmed no read path assumes `output` non-empty when `time.compacted` is set — it's replaced by
  `"[Old tool result content cleared]"` (`message-v2.ts:293`). Leaving `metadata`/`input`/`title`/`time`
  intact is correct and harmless.
- **`attachments`:** OpenCode ALSO drops attachments for a compacted part (`message-v2.ts:296`), so they're
  equally unused post-compaction and would be safe to clear too if you want more reclaim. Your call —
  leaving them is conservative and fine; clearing them is also safe by the same evidence. If you do clear
  them, do it in the same `json_set` transaction and keep valid JSON (e.g. set `$.state.attachments` to
  `[]`, don't delete the key if other code expects an array). Recommendation: start with `output` only
  (biggest win, simplest), add `attachments` later if measured bytes justify it.
- Keep `time.compacted` itself intact (it's the flag OpenCode keys on).

**Q3 — live fd on `.db`/`-wal`/`-shm` as the "in use" signal: CORRECT and authoritative.**
- Single serialized connection per process; Desktop can add a SECOND server process, so process-name alone
  is insufficient — the fd check is right. Confirmed no connection pool.
- **False-positive risk (transient opens):** the DB layer opens the connection as a scoped resource and
  closes it on teardown (`database.ts` `layerFromPath`/finalizer). Short-lived CLIs (`opencode export`,
  `session list`, etc.) open the DB briefly and close it — so a fd check could catch one mid-run and report
  "in use" for a few ms. That's a benign false-positive (fail-safe: you'd just decline to mutate). To reduce
  it: sample the fd set twice a moment apart, or simply treat any live fd as "don't mutate" (safer, which is
  what you want). Don't try to distinguish "transient" from "real" — err toward not mutating.
- Also confirmed: check ALL same-UID processes (Desktop server, TUI-embedded server, `serve`, `web`), not
  just a terminal `opencode`.

**Q4 — `wal_checkpoint(TRUNCATE)` + `VACUUM` as default safe action (backup first, no fd): NO OBJECTION.**
- Confirmed: WAL mode, `synchronous=NORMAL`, `busy_timeout=5000`, no `auto_vacuum`, no `VACUUM` anywhere in
  OpenCode source (`database.ts:27-32`; #31526). Offline checkpoint+VACUUM is pure space reclaim with no
  data-model risk. Backing up the db+wal+shm family first is the right discipline. Good as the default.

**Q5 — anything else unsafe/misattributed?** A few small notes, none blocking:
1. **`event_bloat` estimate is report-only — keep it that way (you do).** Re-confirming: `event` is the
   replay log; deleting superseded `message.updated` rows is unsafe. Your removal of `--compact-events` is
   correct. (One nuance for the ESTIMATE's honesty: "superseded waste" assumes older snapshots are pure
   duplicates; they're full-message snapshots at each update, so the estimate is a reasonable upper bound on
   reclaimable-IF-it-were-safe. Label it as "would-be reclaimable upstream," which you do.)
2. **`quick_check` on a `mode=ro` connection while OpenCode writes** can occasionally report transient
   inconsistencies mid-write (it's reading a live WAL DB). Treat a single non-"ok" from a running DB as
   NOTICE-worth-rechecking-when-stopped, not an immediate WARN of corruption. When OpenCode is stopped,
   `quick_check` is authoritative.
3. **Discovery: also handle `OPENCODE_DB=:memory:` / absolute** — your D-1 says "`$OPENCODE_DB`'s parent";
   note `:memory:` has no parent (skip DB-file checks) and an absolute `OPENCODE_DB` may point OUTSIDE the
   data dir (so the DB family and the snapshot/log/repos subtrees can live in different roots). Resolve the
   DB path and the data-dir subtrees independently rather than assuming they share a parent.
4. **`opencode-<channel>.db` glob:** `opencode*.db` is right, but it will also match your own
   `opencode-db-cleanup-*` backups if they ever land in the data dir — scope the glob to the data dir top
   level and exclude your backup name prefixes so `doctor` doesn't misreport a backup as the live DB.
5. **`/tmp/*.so` origin** remains inferred (Bun-extracted native/wasm; no OpenCode source writes them). Your
   REPORT-default + mmap/fd-guarded + `--force` on non-Linux is the right posture. The empirical `lsof`
   confirmation you noted is worth doing once before shipping the delete path.
