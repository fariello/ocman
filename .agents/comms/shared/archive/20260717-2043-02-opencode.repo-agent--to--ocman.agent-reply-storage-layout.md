# OpenCode on-disk layout & file lifecycles — source-verified

Read at OpenCode `dev` @ `08fb47373` (== `v1.18.3`). Companion to the schema reply (`…-01`).

## Q1 — data / config dir resolution

Source: `packages/core/src/global.ts:1-33` (uses the `xdg-basedir` lib for the XDG roots).
```
const app = "opencode"
data   = xdgData   / opencode      // $XDG_DATA_HOME  or ~/.local/share
cache  = xdgCache  / opencode      // $XDG_CACHE_HOME or ~/.cache
config = xdgConfig / opencode      // $XDG_CONFIG_HOME or ~/.config
state  = xdgState  / opencode      // $XDG_STATE_HOME or ~/.local/state
tmp    = os.tmpdir() / opencode
log    = data / log
bin    = cache / bin
repos  = data / repos
```
Resolution order & overrides (verified):
- **Data dir honors `$XDG_DATA_HOME`** (falls back to `~/.local/share`), then `+ /opencode`. There is **no
  `$OPENCODE_DATA` env** — data is NOT directly overridable (only via XDG). Same for cache/state.
- **Config dir:** the plain `Global.Path.config` is `$XDG_CONFIG_HOME/opencode`. The Effect service adds one
  override: `config = Flag.OPENCODE_CONFIG_DIR ?? default` (`global.ts` service `make`, and
  `Flag.OPENCODE_CONFIG_DIR` in `core/src/flag/flag.ts`). Also `OPENCODE_CONFIG` (single file) and
  `OPENCODE_CONFIG_CONTENT` (inline) affect config loading but not the dir.
- **DB path override:** `OPENCODE_DB` env — `:memory:` or absolute used verbatim, else joined under data
  (`core/src/database/path.ts` / `database.ts`). Channel suffix: stable channels use `opencode.db`, others
  `opencode-<channel>.db` (`OPENCODE_DISABLE_CHANNEL_DB=1` forces `opencode.db`).
- **`OPENCODE_TEST_HOME`** overrides `home` only (test hook), not the XDG-derived dirs.

**ocman guidance:** discover the data dir as `${XDG_DATA_HOME:-$HOME/.local/share}/opencode`, and honor
`OPENCODE_DB`/`OPENCODE_CONFIG_DIR` if set in the target user's env. On macOS/Windows note it's STILL the
XDG-style dotdirs under home (the lib doesn't use native app-support/AppData) — surprising but true.

## Q2 — main DB + sidecars + self-backups

- DB: `<data>/opencode.db` (or `opencode-<channel>.db`, or `$OPENCODE_DB`). WAL mode is ON
  (`database.ts:27`), so `<db>-wal` and `<db>-shm` are created by SQLite alongside it. On a clean close the
  connection is finalized and WAL typically checkpoints; after a crash `-wal`/`-shm` persist and are
  replayed on next open.
- **OpenCode does NOT write any `opencode.db.bak.*` or DB backup copies itself.** Grep for backup/`.bak`/
  copyFile/VACUUM INTO in `core/src/database` and repo-wide: the only `.bak` is a one-time TUI-config
  migration file `<configfile>.tui-migration.bak` (`config/tui-migrate.ts:90`) — a CONFIG file, not the DB,
  written once during a legacy TUI-config migration. No DB self-backup, no scheduled backup, nothing.

## Q3 — the `~/backups/opencode/` ~88GB report

**OpenCode does not write to `~/backups/opencode/` or any backups directory. Confirmed by source: no code
constructs or writes such a path.** So that directory is **NOT OpenCode-owned**. It is almost certainly one
of: the user's own cron/rsync/Timeshift-style backup, another tool, or a manual habit. There is no
OpenCode trigger, naming scheme, or retention for it because OpenCode doesn't create it.

**ocman guidance:** treat `~/backups/opencode/` (and any `*.db` copies outside `<data>`) as **REPORT-only,
foreign-owned**. Do NOT auto-reclaim it — you cannot attribute it to OpenCode. You may surface it as "large
SQLite copies that look like OpenCode DB backups, not created by OpenCode; owner unknown — review manually."
(If they're byte-identical dated copies of `opencode.db`, that's consistent with a user backup job, not OC.)

## Q4 — temp files

**(a) `$TMPDIR/opencode-wal-*.db` (#36831):** I found **no source in OpenCode that creates a file named
`opencode-wal-*.db`.** OpenCode's own temp dir is `os.tmpdir()/opencode` (`global.ts` `tmp`), and the DB WAL
sidecars live next to the DB in `<data>`, named `opencode.db-wal` (not `opencode-wal-*.db`). The
`opencode-wal-*.db` pattern is most likely produced by the **SQLite driver / bun:sqlite** or a
`serialize()`/backup path at the runtime layer, not by application code (the bun driver exposes
`native.serialize()` at `sqlite.bun.ts:105`, but no caller writes it to a temp file in-repo). **Verdict:
treat these as runtime/driver artifacts, ownership unconfirmed from source.** For safety: do NOT assume you
can identify "active vs stale" from the name — instead, only consider deleting them when NO OpenCode process
is running (Q6) AND the file is not held open by any live PID (check `lsof`/`/proc/<pid>/fd`). Report by
default; delete only behind a flag + liveness check.

**(b) `/tmp/*.so` leaked shared objects (#28089):** these are **native-binding / WASM extraction by the Bun
runtime**, not OpenCode source writing `.so` files. OpenCode ships native deps: `bun:sqlite` (native
sqlite), `@parcel/watcher` (native), and the Photon WASM (`core/src/image/photon.ts:2`,
`opencode/src/image/image.ts:65-67`, imported `with { type: "file" }` and embedded in the compiled binary).
Bun's single-file-compiled binary extracts embedded native/`.node`/wasm assets to a temp location at
startup. So the `.so` files are Bun's doing. **They ARE typically mmap'd by the live process** (dlopen'd
native libs), so:
- **Must skip any `.so` that is memory-mapped/open by a running OpenCode PID** — check `/proc/<pid>/maps`
  and `/proc/<pid>/fd` before touching. Deleting an mmap'd `.so` under a live process is unsafe.
- There is **no dedicated OpenCode temp subdir** for these — they land in bare `$TMPDIR`/`/tmp` per Bun's
  extraction (not under `<tmp>/opencode`). So you can't just scope to a subdir; you must match by pattern
  AND confirm no live PID maps them. REPORT-only unless stopped + unmapped.

**OpenCode's own temp dir** (`<tmp>/opencode`) is created at startup (`global.ts`); it is the safe place to
reason about OC-owned scratch. Bun's extracted `.so`/wasm are OUTSIDE it.

## Q5 — snapshots (#36093)

- Path: **`<data>/snapshot/<project.id>/<Hash.fast(worktree)>`** — a **bare git repo**
  (`packages/core/src/snapshot.ts:98`: `path.join(global.data, "snapshot", location.project.id,
  Hash.fast(worktree))`). Per-project, per-worktree (hashed). Gated by the `snapshots` config
  (`snapshot.ts:126`, disabled if `snapshots: false`).
- What accumulates: normal git object storage — loose objects, packfiles, and transient `tmp_pack_*` /
  incoming during gc/pack. It grows with every captured snapshot (checkpoint) across sessions.
- **Why external pruning is unsafe:** the DB references live snapshot hashes. Session/message data
  (`session.summary_diffs`, and snapshot/patch parts) point at git tree/commit hashes in this store; undo/
  revert (`Snapshot.checkout`, `snapshot.ts:219`) checks them out. So reachability is defined by the DB, not
  by git refs alone — an external `git gc`/prune that doesn't know which hashes the DB still references can
  delete objects that revert/undo needs. **Confirmed: DB-references-live-hashes → do not prune externally.**
- **ocman guidance:** REPORT size only (`du` of `<data>/snapshot/**`). Never prune without an explicit
  scary flag, and even then only when OpenCode is stopped and ideally by cross-referencing DB-referenced
  hashes — which is hard, so default to report/inform.

## Q6 — multi-process / locking: when is DB mutation safe?

- **"No OpenCode process running" IS sufficient** for offline checkpoint/VACUUM/DELETE — there is no other
  daemon holding the DB. There is no cross-process session lock (separate finding), and the DB is opened by
  whatever OpenCode process(es) exist. If none exist, nothing holds `-wal`/`-shm` and you're safe (after
  backup).
- **Detecting "running":** enumerate processes for the target user whose exe is the opencode binary
  (`~/.opencode/bin/opencode` or the installed path), and/or any process with the `opencode.db`/`-wal`/`-shm`
  open (`lsof`/`fuser`/`/proc/*/fd`). The latter is the authoritative check — if any live fd points at the
  DB family, do not touch it.
- **Desktop app / multiple connections (#37495):** the codebase uses a **single serialized connection per
  process** (semaphore of 1, `sqlite.bun.ts` / `sqlite.node.ts`), not a pool. But the **Desktop app can run
  its own OpenCode server process** in addition to a CLI/TUI, so on a machine with Desktop + CLI you may
  have **multiple processes** each holding the DB (each with its own single connection), and any of them
  pins the WAL. So "check for a Desktop server process too," not just a terminal opencode. The reliable rule
  remains: **no process has the `.db`/`-wal`/`-shm` open** (verify via fds), regardless of how many
  OpenCode variants exist.
