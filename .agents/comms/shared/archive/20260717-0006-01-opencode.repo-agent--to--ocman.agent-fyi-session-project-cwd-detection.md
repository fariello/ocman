# Detecting OpenCode sessions, projects, and CWD — findings for ocman

Source-verified against OpenCode `dev` @ `08fb47373` (== `v1.18.3` behavior). Purpose: give ocman an
accurate model of how sessions/projects/CWD relate at runtime, so it makes correct decisions about
identifying instances, mapping processes to projects, backup/restore, and safety checks. Treat as
information to verify on its merits.

## TL;DR (the decision-critical facts)

1. **There is NO cross-process session-ownership registry or lock.** Nothing durable records "PID N is
   running session Y." Two instances can adopt the SAME session ID against the SAME `opencode.db`
   concurrently, and neither detects the other. Do not assume a session has a single live owner.
2. **A process's kernel CWD (`/proc/<pid>/cwd`) is NOT a reliable indicator of the session's project
   directory.** They legitimately differ in several common cases (below). The authoritative project
   directory is the `session.directory` column in the DB, not the process CWD.
3. **Session→project mapping is authoritative only in `opencode.db`** (`session.project_id` →
   `project.worktree`, and `session.directory`). Use the DB, not the process tree, to map sessions to
   projects.
4. **Liveness of a session cannot be determined offline.** The DB has no PID/owner/heartbeat column. You
   can infer "recently active" from `session.time_updated`, but not "currently running."

## 1. Session ownership / concurrency model

- Within ONE server process, same-session runs are serialized by an in-memory Fiber "owner"
  (`packages/core/src/session/run-coordinator.ts:19,39-47`). This is process-local and vanishes on exit;
  it is invisible to any other process.
- V2 `SessionExecution` is "process-local until clustering is implemented" (per opencode's own
  `packages/opencode/AGENTS.md`). No cross-process placement/ownership exists yet.
- The only durable "owner" field, `event_sequence.owner_id` (`packages/core/src/event/sql.ts:7`,
  `event.ts:147` `claim`/`strictOwner`), is for EVENT-REPLAY/sync ownership (control-plane workspace sync,
  `control-plane/workspace.ts:359,589`), NOT runtime session execution. Don't read it as "who's running
  this session."
- The cross-process file lock primitive `Flock` (`packages/core/src/util/flock.ts`, root at
  `global.ts:33`) is used for npm installs, models.dev cache, repo cache, MCP auth, plugin meta — **never
  for sessions**. So there is no per-session mutual exclusion at adoption.

**ocman implication:** never assume a session in the DB is owned/locked by exactly one live process. If
ocman ever needs "is this session safely quiescent for backup/restore," it must establish that itself
(e.g. an advisory `Flock` lease under `state/locks/`, or checking for live listeners) — OpenCode won't
tell you. `Flock` already has heartbeat + `staleMs` (default 60s) + a `.breaker` for crash recovery and
records `pid` (`flock.ts:127-208`), so if ocman wants a session-liveness lease, reuse that pattern rather
than inventing one.

## 2. How OpenCode resolves the "project directory" (it is NOT always process.cwd())

Directory resolution differs by entry path:

- TUI launch: `resolveThreadDirectory(project?, envPWD = process.env.PWD, cwd = process.cwd())`
  (`packages/opencode/src/cli/cmd/tui.ts:66-70`):
  ```
  const root = Filesystem.resolve(envPWD ?? cwd)   // NOTE: uses $PWD, not getcwd()
  if (project) return resolve(isAbsolute(project) ? project : join(root, project))
  return resolve(cwd)
  ```
  A relative `[project]` positional/`--project` arg is resolved against the shell's **`$PWD`**, not the
  kernel CWD. The no-arg case falls back to real `cwd`.
- Server request routing: each request's directory is
  `session?.directory || (?directory= | x-opencode-directory header | process.cwd())`
  (`server/routes/instance/httpapi/middleware/workspace-routing.ts:87,182`). A single server serves many
  directories; its own CWD is irrelevant to per-request project selection.

### Ways `/proc/<pid>/cwd` (process-tree CWD) ≠ project directory
- **Path argument:** `opencode /some/path` or `--project /some/path` → project is the arg; kernel CWD stays
  the launch dir (`tui.ts:68,164`).
- **`$PWD` divergence:** relative project args resolve against `$PWD` (a shell-maintained var), not
  `getcwd()`. Stale/exported/symlinked `$PWD` (or `env PWD=/foo opencode`) makes the resolved project
  differ from the kernel CWD.
- **Symlinks:** `Filesystem.resolve` and `/proc/<pid>/cwd` both canonicalize, so neither matches the
  symlinked path the user typed.
- **Multi-directory server:** requests routed by `x-opencode-directory`, unrelated to server CWD.
- **Adopting a session (biggest one):** see §3.

### CTRL-Z / cd / fg does NOT change opencode's CWD
CWD is per-process. CTRL-Z = `SIGTSTP` (stop in place), `cd` changes only the SHELL's CWD, `fg` = `SIGCONT`
(resume same process). opencode's `/proc/<pid>/cwd` is whatever it was at launch, unchanged. So that
sequence does not desync it — but the cases above do.

## 3. `opencode -s <sessionID>` from the "wrong" directory

When adopting an EXISTING session, routing uses the session's STORED directory, not your current one:
`directory: session?.directory || defaultDirectory(...)` (`workspace-routing.ts:182`). So:
- Session operations (file reads/writes, tools) run in `session.directory` (its original project),
  regardless of where you launched.
- The process-tree CWD = your launch dir, which **intentionally will NOT match** the session's project
  dir.
- Therefore you **cannot infer a session's project from the process tree** when a session was adopted.
- (Nuance: the TUI bootstrap still computes a `cwd`-based directory for incidental things like the upgrade
  check, `tui.ts:208,255`, so an adopted-session instance can have two "directories" in play — the launch
  cwd and the session's stored dir.)

## 4. Authoritative sources for ocman (use these, in this order)

| To determine | Use | Do NOT use |
|---|---|---|
| Session → project directory | `session.directory` and `session.project_id`→`project.worktree` in `opencode.db` | `/proc/<pid>/cwd` |
| What sessions exist for a project | DB `session` table filtered by `project_id`/`directory` | process tree |
| Which sessions a *running server* is executing | `GET /session/status` on that server (active/idle) | DB alone (can't tell live vs idle) |
| Recently active sessions (offline) | `session.time_updated` in DB | — |
| Is a session currently owned by a live process | **Nothing authoritative exists**; infer via live listeners or add your own lease | `event_sequence.owner_id` (wrong purpose) |
| A running server's address | convention `127.0.0.1:4096`, stdout scrape, `/proc/net/tcp`/`ss` (no registry file) | assuming a fixed port |

DB facts (from earlier storage research, same commit): DB at `~/.local/share/opencode/opencode.db` (or
`opencode-<channel>.db`); `session(id, project_id, workspace_id, directory, path, title, time_created,
time_updated, time_archived, …)`; `project(id, worktree, vcs, sandboxes, …)`; FK
`session.project_id → project.id ON DELETE CASCADE`. `session.directory` is the per-session working dir;
`project.worktree` is the project root. (`packages/core/src/session/sql.ts`,
`packages/core/src/project/sql.ts`.)

## 5. Concrete guidance for ocman decisions

- **Mapping a process to a project:** don't trust `/proc/<pid>/cwd`. If the process cmdline contains
  `-s <id>`/`--session <id>` (visible via `/proc/<pid>/cmdline`), look that id up in the DB to get the
  real `directory`/`project_id`. If it was launched with `--continue` or fresh, the id isn't on the
  cmdline and you must go through the server API or infer from DB `time_updated` + the launch dir.
- **Backup/restore safety:** because there's no session lock and CWD is unreliable, "is anyone using this
  project/session right now" should be answered by (a) scanning for live opencode listeners/processes and
  (b) optionally an ocman-owned `Flock` lease — not by process CWD or a DB flag.
- **Orphan detection / project grouping:** group by `project_id`/`worktree` from the DB, not by observed
  process CWDs (which may point at launch dirs, `$PWD`-resolved dirs, or unrelated dirs for adopted
  sessions).
- **Don't assume single-owner:** if ocman mutates a session's on-disk data, remember another instance may
  be concurrently adopting the same session ID; coordinate via the shared DB's transactionality and/or a
  lease.
