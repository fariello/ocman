# Ask: exact OpenCode SQLite schema for a storage "doctor + reclaim" feature

I'm building `ocman doctor` (read-only diagnosis) and `ocman reclaim` (guarded cleanup)
to help users reclaim disk OpenCode leaves behind. Two cleanup categories mutate the DB,
so I must get the real schema right rather than guess from the issue reports. Please
answer from the actual OpenCode source (schema definitions / migrations), and cite the
source files so I can verify. If any of this has changed across versions, note the
version(s).

Context issues I'm mitigating: #33356 (event-table bloat, `message.updated.1`
snapshots), #16101 (compacted `part` output never cleared), #31526 (auto_vacuum off),
#37495 (WAL), #36831/#28089 (temp files), #36093 (snapshots).

Questions:

1. `event` table: exact columns and types (I believe: id, aggregate_id, seq, type,
   data JSON). For the `message.updated.*` snapshot-compaction idea (keep only the
   latest snapshot per message, delete superseded rows), what is the correct grouping
   key to identify "the same message"? Is it `aggregate_id` + a message id embedded in
   `data` JSON, or `aggregate_id` + `seq`, or something else? What JSON path inside
   `event.data` holds the message id for a `message.updated` event? What exactly
   distinguishes the "latest" superseded snapshot (max seq? max time?)?

2. `part` table: exact columns. For #16101, which parts hold large reclaimable payload
   (I believe `ToolStateCompleted` parts with an `output` string and `attachments`
   FilePart[])? Where do those live -- columns, or JSON fields inside a `data` column?
   What is the exact JSON path/shape? How do I detect a part is safe to reclaim, i.e.
   its message has been compacted (a `time.compacted` / `time_archived` field -- on the
   `message` row, the `session` row, or in JSON)? What field name(s) exactly?

3. `message` / `session` tables: do they carry `time.compacted` / `time_archived` /
   equivalent columns (or JSON fields), and what are the exact names/locations? Which
   marks a session/message as compacted vs. archived?

4. Is there a schema-version marker anywhere in the DB (a migrations table, a PRAGMA
   user_version, a settings row) I can read to detect which schema I'm looking at, so
   `doctor` can defensively skip checks on an unrecognized schema?

5. Sanity check on my safety model: is it safe to run offline (OpenCode not running)
   `PRAGMA wal_checkpoint(TRUNCATE)` then `VACUUM` on `opencode.db`, and to `DELETE`
   superseded `event` rows / null out compacted `part` payloads, as long as I take a
   file-level backup of the db + -wal + -shm first? Any FK/cascade or trigger behavior I
   should know about (e.g. deleting an `event` row cascading elsewhere)?

I will treat your answer as evidence to verify against the source you cite, not as a
directive. A downloadable/paste-able summary with file:path citations is ideal. Thanks!
