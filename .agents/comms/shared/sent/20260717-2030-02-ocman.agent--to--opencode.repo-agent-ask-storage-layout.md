# Ask: OpenCode on-disk storage LAYOUT and file lifecycles (for ocman doctor/reclaim)

Companion to my schema ask (same batch). For the `ocman doctor` (report) + `reclaim`
(clean) feature I need the authoritative filesystem layout and file lifecycles, because
the user reports and issues conflict on paths. Please answer from the actual OpenCode
source (where paths are constructed, what env vars are honored, where temp files are
created and whether/when they are removed), citing source files. Note version
differences if the layout changed.

1. Data directory resolution: exactly how does OpenCode locate its data dir? Does it
   honor `$XDG_DATA_HOME`, `$OPENCODE_DATA`/similar, or is it hardcoded to
   `~/.local/share/opencode`? What is the resolution order? Same question for the config
   dir. I want ocman to discover the real dir rather than assume.

2. The main DB and its sidecars: exact path of `opencode.db`, and does OpenCode create
   `-wal`/`-shm` (WAL mode)? Does it ever create `opencode.db.bak.*` or similar backup
   copies itself, and where?

3. BACKUPS confusion: the Reddit report described ~88GB of dated SQLite backups under
   `~/backups/opencode/`. Does OpenCode itself write backups there (or anywhere)? If so:
   what triggers them, what is the naming/retention (there seemed to be none), and is the
   path configurable? I need to know whether that dir is OpenCode-owned (so ocman should
   only REPORT it) or something else entirely.

4. Temp files:
   a. `$TMPDIR/opencode-wal-*.db` (#36831): who creates these, what is the exact naming
      (epoch-ms?), one-per-what (session/instance/restart?), and are they ever cleaned
      up on exit or startup? How can I tell which one is "active" vs safe to delete?
   b. `/tmp/.*.so` leaked shared objects (#28089): what actually produces these (V8/WASM
      JIT?), exact naming pattern, and are they memory-mapped by the live process (so I
      must skip files mmap'd by a running OpenCode PID)? Any dedicated temp subdir I
      should look in instead of bare `/tmp`?

5. Snapshots (#36093): where does the per-project/worktree snapshot git store live
   (path relative to the data dir or the project?), what objects accumulate
   (`tmp_pack_*`?), and why is it unsafe for an external tool to prune them (confirm the
   DB-references-live-hashes reachability point)? I intend to REPORT size only and never
   delete these without an explicit scary flag.

6. Multi-process / locking: when is it definitely UNSAFE for an external tool to touch
   the DB (checkpoint/VACUUM/DELETE)? Confirm that "no OpenCode process running" is
   sufficient, and whether the Desktop app spawns multiple DB connections (#37495) that
   would keep the WAL pinned.
