## Goal / motivating scenario

A network/SSH interruption terminates the terminal, but the `opencode` process it was running
keeps running orphaned (detached from the dead tty), indefinitely. On reconnect the user
currently must: `cd` back to the project, find and kill that stale opencode, then relaunch
opencode on the session it was on (they prefer `-s ses_XXXX`, visible in `ocman ls --long`).

`ocman reconnect` automates the annoying middle+end: from the current directory, find the
opencode running here, kill it (if unambiguous, else ask), and foreground-exec a fresh
`opencode -s ses_XXXX` in the user's reconnected shell. The user still does the `cd`.

## Design decisions (settled with maintainer)

- **Command:** top-level `ocman reconnect` (names the SSH-reconnect use case). Combined
  kill+relaunch only; a standalone kill is deferred to a future IPD.
- **Target selection (current dir):** find own-user opencode instances whose `cwd` is AT or
  UNDER the current directory (reuse the `lr` detection + filter path).
  - exactly one -> kill it (SIGTERM);
  - zero -> nothing to kill; proceed to launch;
  - many -> list them and ASK interactively, offering individual picks AND an "all" option;
    kill the chosen set.
- **Session to resume:**
  - single kill -> the killed process's `-s ses_XXXX` if its argv had one (via
    `_attribute_session` provenance `launched-with`), else the most-recently-updated session in
    the cwd's project. Show which it picked and why.
  - kill-all (many case) -> ASK which session to resume (from the project's sessions).
  - zero kill -> most-recent session in the project; if ambiguous/none, ASK (or error clearly
    if the project has no sessions).
- **Relaunch model:** FOREGROUND `os.execvp("opencode", ["opencode", "-s", ses_id])` in the
  user's current shell, inheriting its environment. ocman BECOMES opencode (point of no
  return). No daemon; this sidesteps the lost-environment / lost-server-password risk of a
  detached respawn because the new process inherits the reconnected interactive shell's env.
- **Kill/exec timing:** SIGTERM, then POLL `/proc/<pid>` up to a short timeout (~5s) for exit.
  If it exits -> exec opencode. If it does NOT exit in time -> STOP and report (suggest
  re-running, a future `--force` for SIGKILL); do NOT exec over a still-live instance.
- **Confirmation:** confirm by default (show kill target(s) + the exact launch command); `-y`
  / `--yes` skips; `--dry-run` prints the plan and does nothing (no signal, no exec).
- **Platform:** Linux-only (detection is Linux-only; `os.execvp` is POSIX). Same graceful
  "Linux only / NOT an all-clear" degradation as `lr` on non-Linux; never a silent no-op.

## Project conventions discovered (Step 0)

- Guiding principles: universal fallback. No em/en dashes in authored prose. Plans lifecycle
  `.agents/plans/pending/` -> `executed/`; `YYYYMMDD-HHMM-NN-<slug>.md`.
- Contract: path-scoped commits, never push without approval, paste REAL pytest output.
- Detection: `detect_running_instances` (cli.py:8048) enriches `detect_running_opencode_status`
  (cli.py:7775). Per-instance fields include `pid`, `ppid`, `user`, `cwd`, `cmdline`,
  `project`, `kind`, `listeners`, `auth`, `exposed`, `vulnerable`, and `session`
  (`_attribute_session`, cli.py:8104). Detection is HARD Linux-gated (win32/non-linux ->
  "unknown"/raise `RunningDetectionError`); cwd from `/proc/<pid>/cwd`.
- `_attribute_session` (cli.py:8104-8130) ALREADY extracts `-s`/`--session ses_XXXX` from the
  captured cmdline (regex at 8111) and returns provenance `launched-with (may be stale)`. This
  is the reliable "known session" path; otherwise provenance is `directory (one-to-many)` or
  `unknown`.
- Ownership: enumerator defaults to current user (`ps -u`); authoritative own-ness is
  `os.stat(f"/proc/{pid}").st_uid == os.getuid()` (cli.py:8075-8082). ocman excludes its own
  pid/ppid from the listing (7810-7814).
- No process-signalling code exists anywhere (only ocman's OWN SIGINT/SIGTERM handlers for temp
  cleanup at 4063). No `os.kill`, no Popen of a long-running opencode. `require_opencode()`
  (cli.py:1226) validates `shutil.which("opencode")` presence only (returns None).
- `_current_user` (7736), `_render_running_opencode` (7857) for listing, `confirm_destructive`
  (11589) for typed-yes, `cli_list_running` (11947) as the sibling read command + its `pattern`
  filter closure.
- `require_safe_to_mutate` is NOT used here: it refuses to act WHILE opencode runs, which is the
  opposite of reconnect's purpose. Reconnect deliberately acts on a running instance.

## Deferred / out of scope

- **Standalone kill (no relaunch)** / `--no-launch`: deferred to a future IPD (maintainer chose
  combined-flow-now).
- **SIGKILL escalation / `--force`:** out of scope; if SIGTERM does not reap within the timeout,
  reconnect stops and reports. (A future `--force` can add SIGKILL with its own confirm.)
- **macOS/Windows:** out of scope (detection is Linux-only). macOS exec would work but detection
  does not; revisit only if detection gains a macOS path.
- **TUI reconnect affordance:** out of scope (CLI-only here).
- **Preserving the killed process's full argv/env** (flags beyond `-s`, model, server password):
  out of scope and intentionally NOT attempted. Reconnect launches a clean `opencode -s <id>`
  inheriting the current shell's env; the IPD documents that other original flags/env are not
  reproduced.

## Anti-regression / invariants

- No existing command changes; purely additive (new top-level verb + new helpers).
- Detection/ownership semantics unchanged; reconnect is a new consumer of `detect_running_instances`.
- Never signals a non-own-user pid; never signals a pid that no longer looks like opencode
  (PID-reuse guard, best-effort per RC-14); never execs over a still-live instance.
- `--dry-run` AND a user "no" at the single confirm each perform ZERO side effects (no signal,
  no exec) and exit 0 (RC-10).
- `require_opencode()` is checked BEFORE any kill; a missing binary never leaves the user with a
  killed instance and no relaunch (RC-11).
- A multi-kill that only partially succeeds STOPS (no exec) and reports which pids died vs
  survived (RC-12); reconnect never execs after a partial kill.
- If there is no session to resume for the CWD, reconnect errors clearly and NEVER execs a bare
  `opencode` with no `-s` (RC-13).
- ocman never signals its own pid/ppid (already excluded from the listing); re-assert in Step 1.

## Post-execution summary

  `detect_running_instances(all_users=False)`, re-checks own-user via
  `os.stat(/proc/<pid>).st_uid == os.getuid()`, keeps instances whose cwd == or is under CWD.
- Step 2: `_kill_pid_gracefully(pid, timeout=5)` + `_pid_looks_like_opencode` + a new
